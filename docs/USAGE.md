# git-gui 使用指南

git-gui 有两种形态:**桌面 GUI**(产品主体)与**终端 TUI**。二者共享同一个零依赖核心库 `gitcore`,因此安全更新、冲突魔法棒、崩溃恢复、rerere 等核心能力两者一致。本指南先讲 GUI,再讲 TUI,最后集中说明这些共享的核心概念。

---

# 一、桌面 GUI

## 启动与打开项目

```bash
# 开发模式(自动开窗口)
cd gui && npm install && npm run tauri dev
# 或打包后直接运行安装好的应用
```

启动进入**项目选择页**:

- **最近打开**:点选历史项目(可移除)。
- **选择目录…**:打开本地任意 git 仓库。
- **克隆仓库…**:输入远程 URL + 目标父目录,带进度条、可取消;完成后自动打开并加入最近列表。
  - 私有库走系统 git 凭据;若 git 触发交互式密码提示,GUI 无法捕获会失败,请先在终端配置好凭据。

## 主界面布局

```
┌─ 顶栏 ─────────────────────────────────────────────┐
│ 项目名▾  ↻      [全部更新][全部推送][⋯ 更多]    [⚙] │
├─ 标签栏 ───────────────────────────────────────────┤
│  Changes  │  History                                │
└────────────────────────────────────────────────────┘
```

- **顶栏左**:项目名(点击切换 / 打开其它项目)· `↻` 刷新所有仓库(含子仓库的外部改动)。
- **顶栏右**:`全部更新` · `全部推送` · `⋯ 更多`(更新子仓库 / Stash / Tags)· `⚙` 设置。
- 有新版本时,顶部显示一条**新版本提醒**(可查看更新或忽略此版本)。
- **标签栏**:`Changes`(改动与提交)/ `History`(历史与拓扑图)。

## Changes:暂存与提交

每个仓库一个区块,顶部显示**分支名**(点击切换分支)、领先 / 落后远程的提交数,以及该仓库的更新 / 推送入口。改动以**文件树**呈现,分三个区:**已暂存 / 未暂存 / 冲突**。

- 点文件 → 右侧显示结构化彩色 **diff**。
- 在 diff 中可按**文件 / hunk(变更块)/ 单行**粒度**暂存、取消暂存、丢弃**;行内以字符级 diff 高亮具体改动。
- 未跟踪文件作为"全新增"展示,可直接暂存(等同 `git add`)。
- 填写提交信息 → **提交**(多仓库时对所有有暂存改动的仓库统一提交);支持修正上次提交(amend,仅作用主仓库)。是否跳过 git 钩子由**设置**控制。
- 工作区无改动但本地领先远程时,提交区提供直接**推送**入口。

## 更新:安全拉取(安全网)

- 顶栏 **全部更新**:更新主仓库,并把各子仓库拉取到各自远程最新。
- 单个仓库区块内的 **更新**:只更新该仓库。
- `⋯ 更多 → 更新子仓库`:未初始化的子仓库执行 `init`,已初始化的在**各自当前分支**上更新(留在原分支)。

更新流程全自动:**autostash → fetch(fast-forward 优先)→ 按全局策略 merge / rebase → 还原改动**;有进度显示、可取消(取消发生在 stash 之前,工作区不受影响)。**只有产生冲突时才停下**,自动进入冲突解决。整合策略在设置里配置一次,全局生效。

## 冲突解决

整合冲突时弹出三栏视图:

```
   Ours      │     Base      │    Theirs
  (本地改动)  │  (共同祖先)    │   (远端改动)
```

- **魔法棒**自动定夺只有单边改动的内容,真冲突(两边都改了同一处)才留给你处理。
- 逐块选择保留 **ours / theirs / base**,或**手动编辑**结果。
- 全部解决后**写回**并继续整合;也可随时**放弃**,回退到更新前的状态。

## History:历史与拓扑图

- **单仓库**:左侧是 SVG 彩色**分支拓扑图** + 提交列表。
- **含子仓库**:左侧是主仓 + 各子仓提交**合并后的时间线**,每条带仓库标识。
- 选中提交 → 右侧详情:完整提交信息 + 改动文件 diff(只读)。父仓提交若改了子模块指针,展开成该区间内的子仓提交列表。
- 列表头部有 **reflog** 入口:查看 HEAD 的每一步操作,任意一步可**恢复**(变基 / 重置后的安全网)。

对选中的提交可执行:

- **cherry-pick** / **revert**
- **reset 到此**:soft / mixed / hard(hard 标红并二次确认)
- **打 Tag** / **检出**(进入 detached HEAD)
- **交互式变基(从此提交)**:在编辑器里 reword / squash / fixup / drop / 重排,再开始变基;冲突复用同一套冲突解决流程
- **文件历史** / **blame**:在 diff 中触发,查看单文件的提交历史或逐行作者注解,点 SHA 可跳到对应提交

> 选中子仓库的提交时,以上操作都作用于**该子仓库**。

## 分支管理

点任意仓库的**分支名**打开分支面板:

- 列出 / **切换** / 新建 / 删除本地分支
- **检出远程分支**
- **merge / rebase** 到当前分支
- **重命名**分支
- 从选中提交**新建分支**
- 与工作区**比对 diff** / 与当前分支**对比**(各自独有的提交)
- **smart checkout**:切换被脏工作区挡住时,确认后自动 stash 兜底,改动安全保留

## Stash / Tags / 设置

- `⋯ 更多 → Stash`:创建 / 应用 / 弹出 / 丢弃储藏。
- `⋯ 更多 → Tags`:列出 / 创建(轻量或注释)/ 删除 / 推送 tag 到远程。
- `⚙` **设置**:整合策略(merge / rebase)· 忽略空白差异 · 跳过 git 钩子 · 手动检查更新。

# 二、终端 TUI

同一核心库的终端形态,全屏界面 + 快捷键操作。

```bash
cargo run -p tui          # 在任意 git 仓库目录下运行
```

> 多仓库:同样读取 `.git-gui/repos.toml`,左侧边栏列出所有仓库,`Tab` 切换。

启动后进入 **Status 面板**,显示当前分支 / upstream / 领先落后 / 工作区状态 / 冲突文件。

| 键位 | 功能 | 说明 |
|------|------|------|
| `j` `k` `↑` `↓` | 导航 / 滚动 | 状态文本过长时滚动查看 |
| `s` | Stage | 文件暂存视图 |
| `b` | Branch | 分支列表(创建 / 切换 / 删除) |
| `h` | Stash | 储藏列表(创建 / 应用 / 弹出 / 丢弃) |
| `S` | Submodule | 子仓库列表 |
| `l` | Log | 提交历史 + 分支图 |
| `d` | Diff | 改动查看 / 逐 hunk · 逐行暂存 |
| `p` | Push | 推送(非快进自动整合后重推) |
| `u` | Update | 安全拉取(自动 stash,进度条,`Esc` 取消) |
| `m` | 策略 | 切换 merge / rebase |
| `R` | Resume | 恢复未完成的冲突解决 |
| `r` | Refresh | 刷新状态 |
| `q` | Quit | 退出 |

> 所有视图支持**鼠标滚轮 / 触摸板**:diff / log 内容区纯滚动不移光标,文件树 / 列表则移动选中。需要终端原生选择文本时按住 `Shift` 拖选。

**各视图要点:**

- **Stage(`s`)**:可折叠目录树。`Space` 暂存 / 取消(文件或整目录)· `a` 全暂存 · `d` 回滚(stash 兜底,可找回)· `c` 提交。
- **Log(`l`)**:提交历史 + 蓝色分支图,滚到底自动加载更多;`y` 复制完整 SHA;`Enter` 进双栏详情(左:提交信息 + 文件树,右:diff)。
- **Diff(`d`)**:左栏文件树 + 右栏单文件 hunk diff。右栏 `Space` 选行 / 暂存整 hunk,`s` 暂存选中行,`t` 切换未暂存 / 已暂存视图。
- **Branch(`b`)**:`Enter` 切换 · `c` 创建 · `d` 删除。
- **Stash(`h`)**:`c` 创建 · `a` apply · `P` pop · `d` drop。
- **冲突三栏**:`o` / `t` / `b` 选 ours / theirs / base,`←` `→` 切换选择,`J` `K` 块内滚动,`n` `p` 切换冲突文件,`w` 写回当前文件,`c` 完成整合,`x` 放弃。

# 三、核心概念(两种形态共享)

## 安全更新(autostash)

更新前若工作区有未提交改动,先自动 `stash --include-untracked` 兜底,整合成功后 `apply` + `drop` 还原。**用 `apply`+`drop` 而非 `pop`**:即便还原阶段出问题,stash 仍在、改动不丢。取消发生在 stash 之前,工作区不受影响。

## 行级魔法棒

冲突块拆到**行级**做 diff3 自动定夺:

- **单边改动**(只有一边改了某行)→ 自动采纳(标绿)。
- **真冲突**(两边都改了同一行)→ 保留给人工选择(标黄)。

例如下面的冲突,魔法棒自动处理首尾的单边改动,只剩中间真正冲突的行需要你选:

```
<<<<<<< ours
A           ← 只有 ours 改了这行 → 自动保留
B           ← 两边都改了 → 需人工选择
||||||| base
2
3
=======
2
C           ← 两边都改了 → 需人工选择
D           ← 只有 theirs 改了这行 → 自动保留
>>>>>>> theirs
```

整个算法零外部依赖(自写 Myers LCS + 三版本合并)。

## 崩溃恢复

在冲突解决过程中退出或崩溃,下次打开会检测到未完成的整合(残留 autostash + 冲突文件列表),恢复到冲突视图,之前的选择状态与 autostash 都保留。GUI 自动恢复,TUI 按 `R` 恢复。

## rerere

启用 git rerere:第一次解决冲突时记录解法,下次遇到**相同**冲突自动重放,无需重复手动解决。

---

## 特性总结

- ✅ **零依赖核心**:`gitcore` 只依赖 std + spawn git。
- ✅ **安全 pull**:自动 stash / 还原,不丢改动。
- ✅ **行级魔法棒**:冲突拆到行级,自动定夺单边改动。
- ✅ **可视化历史**:彩色拓扑图、三栏 diff、blame、文件历史。
- ✅ **崩溃恢复 + rerere**:中断可续、解法可重放。
- ✅ **多仓库 + 子仓库**:合并提交历史、操作按所属仓库路由。
